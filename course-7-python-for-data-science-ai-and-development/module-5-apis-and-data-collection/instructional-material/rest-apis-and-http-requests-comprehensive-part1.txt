SECTION 1.1: WHAT IS THE HTTP PROTOCOL?
----------------------------------------

DEFINITION
----------
The HTTP protocol can be thought of as a general protocol of transferring 
information through the web. HTTP stands for HyperText Transfer Protocol.

DETAILED EXPLANATION
--------------------
HTTP (HyperText Transfer Protocol) is the foundation of data communication on 
the World Wide Web. It's a protocol, which means it's a set of rules and 
standards that define how information is formatted and transmitted between 
computers over the internet.

WHAT DOES "PROTOCOL" MEAN?
--------------------------
A protocol is like a language or set of rules that two parties agree to use 
when communicating. Think of it like this:

• Protocol = The rules of conversation
• Just like humans need to speak the same language to communicate, computers 
  need to use the same protocol to exchange information
• HTTP is one of many protocols used on the internet
• Other protocols include HTTPS (secure HTTP), FTP (file transfer), SMTP 
  (email), etc.

REAL-WORLD ANALOGY
------------------
Think of HTTP like a postal service:

• HTTP Protocol = The postal service rules and procedures
• Your Request = A letter you send
• The Server = The recipient
• The Response = A reply letter you receive
• The Rules = How to address envelopes, what stamps to use, delivery 
  procedures

Just as the postal service has rules about how to send and receive mail, HTTP 
has rules about how to send and receive information on the web.

KEY CHARACTERISTICS OF HTTP
----------------------------
1. REQUEST-RESPONSE MODEL
   • Client sends a request
   • Server processes the request
   • Server sends a response back
   • This is a one-way communication pattern (client initiates)

2. STATELESS PROTOCOL
   • Each request is independent
   • Server doesn't remember previous requests
   • Each request contains all information needed

3. TEXT-BASED
   • HTTP messages are readable text
   • Easy to debug and understand
   • Can be read by humans (though usually parsed by programs)

4. APPLICATION LAYER PROTOCOL
   • Works on top of TCP/IP
   • Handles high-level communication
   • Doesn't worry about low-level network details

5. FLEXIBLE
   • Can transfer any type of data
   • Commonly used for HTML, JSON, XML, images, videos, etc.

WHAT DOES "TRANSFERRING INFORMATION" MEAN?
------------------------------------------
When we say HTTP transfers information, we mean:

• Moving data from one computer (client) to another (server)
• The client requests information (like a web page)
• The server provides that information
• The information can be:
  - Web pages (HTML)
  - Data (JSON, XML)
  - Images, videos, files
  - API responses
  - Any digital content

HOW HTTP WORKS - THE BIG PICTURE
--------------------------------
1. CLIENT INITIATES REQUEST
   • You type a URL in your browser
   • Browser creates an HTTP request
   • Request is sent to the server

2. SERVER PROCESSES REQUEST
   • Server receives the HTTP request
   • Server understands what is being requested
   • Server finds or generates the requested resource

3. SERVER SENDS RESPONSE
   • Server creates an HTTP response
   • Response contains the requested information
   • Response is sent back to the client

4. CLIENT RECEIVES RESPONSE
   • Browser receives the HTTP response
   • Browser interprets the response
   • Browser displays the content to you

EXAMPLE: VISITING A WEBSITE
----------------------------
When you visit www.example.com:

1. Your browser (client) creates an HTTP request:
   "GET / HTTP/1.1"
   "Host: www.example.com"

2. This request travels over the internet to the server

3. The server at www.example.com receives the request

4. The server processes it and creates a response:
   "HTTP/1.1 200 OK"
   "Content-Type: text/html"
   "<html>...</html>"

5. The response travels back to your browser

6. Your browser displays the HTML page

WHY IS HTTP IMPORTANT?
----------------------
HTTP is crucial because:

1. UNIVERSAL STANDARD
   • Almost all web communication uses HTTP
   • Works across different platforms and devices
   • Supported by all browsers and servers

2. ENABLES WEB BROWSING
   • Without HTTP, you couldn't browse websites
   • It's how web pages are delivered to you
   • It's how you interact with web applications

3. FOUNDATION FOR APIs
   • REST APIs use HTTP
   • Most modern APIs are HTTP-based
   • Enables programmatic access to web resources

4. SIMPLE AND UNDERSTANDABLE
   • Text-based, so it's readable
   • Easy to debug
   • Well-documented standard

SECTION 1.2: HTTP AND REST APIs RELATIONSHIP
---------------------------------------------

DEFINITION
----------
REST APIs function by sending a request, and the request is communicated via 
HTTP message. The HTTP message usually contains a JSON file. HTTP includes 
many types of REST APIs.

DETAILED EXPLANATION
--------------------
REST (Representational State Transfer) is an architectural style for designing 
web services. REST APIs are APIs that follow REST principles, and they use HTTP 
as their communication protocol. This means REST APIs rely on HTTP to send and 
receive data.

WHAT IS A REST API?
-------------------
REST API stands for:
• REST = Representational State Transfer (an architectural style)
• API = Application Programming Interface (a way for programs to communicate)

A REST API is a way for different applications to communicate with each other 
over the internet using HTTP. It follows specific principles:
• Uses HTTP methods (GET, POST, PUT, DELETE, etc.)
• Stateless (each request is independent)
• Uses URLs to identify resources
• Returns data in formats like JSON or XML

THE RELATIONSHIP BETWEEN HTTP AND REST APIs
-------------------------------------------
Think of the relationship like this:

• HTTP = The language/protocol used for communication
• REST = The style/architecture of how to use that language
• REST API = An API that uses HTTP following REST principles

ANALOGY
-------
Think of HTTP and REST APIs like this:

• HTTP = The postal service (the delivery system)
• REST = The format/style of letters (how to write them)
• REST API = A specific way of sending letters using the postal service

You can use HTTP without REST (like regular web pages), but REST APIs always 
use HTTP.

HOW REST APIs USE HTTP
----------------------
REST APIs use HTTP in the following ways:

1. HTTP METHODS
   • GET - Retrieve data
   • POST - Create new data
   • PUT - Update existing data
   • DELETE - Remove data
   • PATCH - Partially update data

2. HTTP URLS
   • REST APIs use URLs to identify resources
   • Example: GET /api/users/123
   • The URL tells the server what resource you want

3. HTTP HEADERS
   • REST APIs use headers for metadata
   • Content-Type: application/json
   • Authorization: Bearer token123
   • Accept: application/json

4. HTTP STATUS CODES
   • REST APIs use status codes to indicate results
   • 200 = Success
   • 404 = Not Found
   • 401 = Unauthorized
   • 500 = Server Error

5. HTTP BODY
   • REST APIs send data in the request/response body
   • Usually in JSON format
   • Example: {"name": "John", "age": 30}

WHAT DOES "HTTP MESSAGE USUALLY CONTAINS A JSON FILE" MEAN?
-----------------------------------------------------------
When we say "HTTP message usually contains a JSON file," we mean:

• JSON (JavaScript Object Notation) = A format for structuring data
• REST APIs commonly use JSON to send data
• The JSON data is placed in the HTTP message body
• Both requests and responses can contain JSON

EXAMPLE: REST API REQUEST WITH JSON
------------------------------------
Here's what a REST API request looks like:

HTTP REQUEST:
-------------
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 45

{
  "name": "John Doe",
  "email": "john@example.com"
}

BREAKDOWN:
----------
• POST /api/users - HTTP method and resource path
• Host: api.example.com - Server address
• Content-Type: application/json - Tells server data is JSON
• Content-Length: 45 - Size of the JSON data
• The JSON object - The actual data being sent

EXAMPLE: REST API RESPONSE WITH JSON
-------------------------------------
Here's what a REST API response looks like:

HTTP RESPONSE:
--------------
HTTP/1.1 201 Created
Content-Type: application/json
Content-Length: 67

{
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com",
  "created": "2024-01-15"
}

BREAKDOWN:
----------
• HTTP/1.1 201 Created - Status line (success, resource created)
• Content-Type: application/json - Response is JSON
• Content-Length: 67 - Size of JSON data
• The JSON object - The actual data returned

WHY DO REST APIs USE JSON?
--------------------------
REST APIs commonly use JSON because:

1. READABLE
   • Easy for humans to read and understand
   • Easy to debug

2. LIGHTWEIGHT
   • Smaller than XML
   • Faster to transmit

3. UNIVERSAL
   • Supported by all programming languages
   • Works across platforms

4. STRUCTURED
   • Can represent complex data structures
   • Supports arrays, objects, nested data

5. STANDARD
   • Widely adopted
   • Well-documented

OTHER DATA FORMATS
------------------
While JSON is common, REST APIs can also use:

• XML (eXtensible Markup Language)
• Plain text
• HTML
• Binary data
• Form data

The Content-Type header tells the server/client what format is being used.

SECTION 1.3: CLIENT-SERVER COMMUNICATION MODEL
-----------------------------------------------

DEFINITION
----------
When you, the client, use a web page, your browser sends an HTTP request to 
the server where the page is hosted. The server tries to find the desired 
resource (by default "index.html"). If your request is successful, the server 
will send the object to the client in an HTTP response, which includes 
information like the type of the resource, the length of the resource, and 
other information.

DETAILED EXPLANATION
--------------------
The client-server model is the fundamental architecture of the web. In this 
model, there are two main roles:

1. CLIENT
   • The requester of information
   • Usually your web browser
   • Initiates communication
   • Sends HTTP requests

2. SERVER
   • The provider of information
   • Stores and serves resources
   • Responds to requests
   • Sends HTTP responses

WHAT IS A CLIENT?
-----------------
A client is:
• A program that requests services or resources
• Usually runs on your computer (like a web browser)
• Initiates communication with servers
• Examples: Chrome, Firefox, Safari, mobile apps, API clients

In the context of HTTP:
• The client is the one making HTTP requests
• The client wants to get something from the server
• The client waits for the server's response

WHAT IS A SERVER?
-----------------
A server is:
• A program that provides services or resources
• Usually runs on a remote computer
• Responds to client requests
• Examples: Web servers, API servers, database servers

In the context of HTTP:
• The server receives HTTP requests
• The server processes requests and finds resources
• The server sends HTTP responses back to clients

REAL-WORLD ANALOGY
------------------
Think of the client-server model like a restaurant:

• CLIENT = You (the customer)
  - You look at the menu (browse websites)
  - You place an order (send HTTP request)
  - You wait for your food (wait for HTTP response)
  - You receive your food (receive HTTP response)

• SERVER = The restaurant (the service provider)
  - Has the menu (has resources)
  - Receives your order (receives HTTP request)
  - Prepares your food (processes request)
  - Serves your food (sends HTTP response)

THE REQUEST-RESPONSE CYCLE
--------------------------
Here's how the client-server communication works:

STEP 1: CLIENT INITIATES REQUEST
---------------------------------
• You type a URL in your browser (e.g., www.example.com)
• Your browser creates an HTTP request
• The request includes:
  - What you want (the resource)
  - Where to send it (the server address)
  - Additional information (headers)

STEP 2: REQUEST TRAVELS TO SERVER
----------------------------------
• The HTTP request travels over the internet
• It goes through routers, switches, and networks
• Eventually reaches the server

STEP 3: SERVER PROCESSES REQUEST
---------------------------------
• Server receives the HTTP request
• Server reads what resource is being requested
• Server looks for the resource (by default "index.html" for root requests)
• Server checks if the resource exists
• Server prepares to send the resource

STEP 4: SERVER SENDS RESPONSE
------------------------------
• Server creates an HTTP response
• Response includes:
  - Status code (200 = success, 404 = not found, etc.)
  - Headers (content type, length, etc.)
  - Body (the actual resource - HTML, JSON, image, etc.)

STEP 5: RESPONSE TRAVELS TO CLIENT
-----------------------------------
• The HTTP response travels back over the internet
• Goes through the same network path
• Eventually reaches your browser

STEP 6: CLIENT PROCESSES RESPONSE
----------------------------------
• Browser receives the HTTP response
• Browser reads the status code
• Browser reads the headers
• Browser reads the body
• Browser displays the content to you

WHAT DOES "BY DEFAULT INDEX.HTML" MEAN?
---------------------------------------
When you visit a website's root URL (like www.example.com), the server doesn't 
know which specific file you want. By convention, servers look for a file 
called "index.html" (or "index.htm", "index.php", etc.) in the root directory.

This is like:
• Going to a library and asking for "a book"
• The librarian assumes you want the main catalog (index)
• They give you the index book by default

EXAMPLES OF DEFAULT FILES:
• index.html - Most common
• index.htm - Alternative HTML extension
• index.php - PHP file
• default.html - Some servers use this
• home.html - Some servers use this

WHAT INFORMATION IS INCLUDED IN THE RESPONSE?
----------------------------------------------
The HTTP response includes several types of information:

1. STATUS CODE
   • Tells if the request was successful
   • Examples: 200 (OK), 404 (Not Found), 500 (Server Error)

2. CONTENT-TYPE HEADER
   • Tells what type of resource it is
   • Examples: text/html, application/json, image/png

3. CONTENT-LENGTH HEADER
   • Tells the size of the resource
   • Example: Content-Length: 1234 (bytes)

4. OTHER HEADERS
   • Date: When the response was sent
   • Server: What server software is being used
   • Cache-Control: How to cache the resource
   • Set-Cookie: Cookies to set

5. RESPONSE BODY
   • The actual resource content
   • HTML code for web pages
   • JSON data for APIs
   • Binary data for images/videos

EXAMPLE: COMPLETE REQUEST-RESPONSE CYCLE
-----------------------------------------
Let's trace through a complete example:

CLIENT REQUEST:
---------------
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html

BREAKDOWN:
----------
• GET /index.html - Requesting the index.html file
• Host: www.example.com - Server address
• User-Agent: Mozilla/5.0 - Browser information
• Accept: text/html - Client accepts HTML

SERVER RESPONSE:
----------------
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1234
Date: Mon, 15 Jan 2024 10:00:00 GMT
Server: Apache/2.4

<!DOCTYPE html>
<html>
<head>
  <title>Example</title>
</head>
<body>
  <h1>Welcome</h1>
</body>
</html>

BREAKDOWN:
----------
• HTTP/1.1 200 OK - Success status
• Content-Type: text/html - It's an HTML file
• Content-Length: 1234 - File is 1234 bytes
• Date: Mon, 15 Jan 2024 10:00:00 GMT - When sent
• Server: Apache/2.4 - Server software
• The HTML content - The actual web page

WHAT HAPPENS IF THE RESOURCE ISN'T FOUND?
------------------------------------------
If the server can't find the requested resource:

SERVER RESPONSE (404 NOT FOUND):
---------------------------------
HTTP/1.1 404 Not Found
Content-Type: text/html
Content-Length: 234

<html>
<body>
  <h1>404 - Page Not Found</h1>
</body>
</html>

The status code 404 tells the client that the resource doesn't exist.

WHAT HAPPENS IF THERE'S A SERVER ERROR?
----------------------------------------
If something goes wrong on the server:

SERVER RESPONSE (500 SERVER ERROR):
------------------------------------
HTTP/1.1 500 Internal Server Error
Content-Type: text/html
Content-Length: 256

<html>
<body>
  <h1>500 - Server Error</h1>
</body>
</html>

The status code 500 tells the client that the server had an error.

SECTION 1.4: HOW HTTP MESSAGES WORK
------------------------------------

DEFINITION
----------
HTTP messages are how information is exchanged between clients and servers. 
There are two types: HTTP requests (from client to server) and HTTP responses 
(from server to client).

DETAILED EXPLANATION
--------------------
HTTP messages are the actual data packets that travel between clients and 
servers. Every interaction on the web involves HTTP messages. Understanding 
their structure is crucial for working with web technologies and APIs.

WHAT IS AN HTTP MESSAGE?
------------------------
An HTTP message is:
• A formatted text document
• Sent between client and server
• Contains instructions and data
• Follows a specific structure

There are two types:
1. HTTP REQUEST - Message from client to server
2. HTTP RESPONSE - Message from server to client

THE STRUCTURE OF HTTP MESSAGES
-------------------------------
Both requests and responses have a similar structure:

1. START LINE (or Request/Response Line)
   • First line of the message
   • Contains key information
   • Different format for requests vs responses

2. HEADERS
   • Additional information
   • Key-value pairs
   • Multiple headers allowed

3. BLANK LINE
   • Separates headers from body
   • Required (even if no body)

4. BODY (optional)
   • The actual data
   • May be empty
   • Contains the resource or data being transferred

VISUAL REPRESENTATION
---------------------
HTTP MESSAGE STRUCTURE:

┌─────────────────────────────────┐
│ START LINE                      │
├─────────────────────────────────┤
│ Header 1: Value 1               │
│ Header 2: Value 2               │
│ Header 3: Value 3               │
├─────────────────────────────────┤
│ (blank line)                    │
├─────────────────────────────────┤
│ BODY                            │
│ (actual content/data)           │
└─────────────────────────────────┘

HTTP REQUEST MESSAGE STRUCTURE
-------------------------------
A request message has this structure:

REQUEST LINE:
-------------
GET /index.html HTTP/1.1

• GET - HTTP method (what action to perform)
• /index.html - Resource path (what you want)
• HTTP/1.1 - HTTP version (protocol version)

REQUEST HEADERS:
----------------
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html,application/json
Accept-Language: en-US

• Host - Server address
• User-Agent - Client software information
• Accept - What content types client accepts
• Accept-Language - Preferred language

REQUEST BODY (optional):
------------------------
{
  "name": "John",
  "email": "john@example.com"
}

• Only present in some requests (POST, PUT, etc.)
• Contains data being sent to server
• Often JSON, XML, or form data

HTTP RESPONSE MESSAGE STRUCTURE
--------------------------------
A response message has this structure:

STATUS LINE:
------------
HTTP/1.1 200 OK

• HTTP/1.1 - HTTP version
• 200 - Status code (result of request)
• OK - Status text (human-readable description)

RESPONSE HEADERS:
-----------------
Content-Type: text/html
Content-Length: 1234
Date: Mon, 15 Jan 2024 10:00:00 GMT
Server: Apache/2.4

• Content-Type - Type of content in body
• Content-Length - Size of body in bytes
• Date - When response was sent
• Server - Server software information

RESPONSE BODY:
--------------
<!DOCTYPE html>
<html>
  <body>
    <h1>Hello World</h1>
  </body>
</html>

• The actual resource/content
• HTML, JSON, images, etc.
• What the client requested

COMPLETE EXAMPLE: HTTP REQUEST
-------------------------------
Here's a complete HTTP request message:

GET /api/users/123 HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: application/json
Authorization: Bearer abc123xyz
Accept-Language: en-US,en;q=0.9

LINE-BY-LINE EXPLANATION:
-------------------------
Line 1: GET /api/users/123 HTTP/1.1
        • GET - HTTP method (retrieve data)
        • /api/users/123 - Resource path (user with ID 123)
        • HTTP/1.1 - Protocol version

Line 2: Host: api.example.com
        • Host header - Tells server which domain
        • Required in HTTP/1.1
        • Allows multiple websites on one server

Line 3: User-Agent: Mozilla/5.0...
        • User-Agent header - Identifies the client
        • Browser or application information
        • Helps server provide appropriate content

Line 4: Accept: application/json
        • Accept header - What content types client accepts
        • application/json means client wants JSON
        • Server can use this to format response

Line 5: Authorization: Bearer abc123xyz
        • Authorization header - Authentication token
        • Bearer token for API authentication
        • Proves client has permission

Line 6: Accept-Language: en-US,en;q=0.9
        • Accept-Language header - Preferred languages
        • en-US preferred, en as fallback
        • Server can use this for localization

COMPLETE EXAMPLE: HTTP RESPONSE
--------------------------------
Here's a complete HTTP response message:

HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 89
Date: Mon, 15 Jan 2024 10:00:00 GMT
Server: nginx/1.18.0
Cache-Control: no-cache

{
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com"
}

LINE-BY-LINE EXPLANATION:
-------------------------
Line 1: HTTP/1.1 200 OK
        • HTTP/1.1 - Protocol version
        • 200 - Status code (success)
        • OK - Status text (human-readable)

Line 2: Content-Type: application/json
        • Content-Type header - Type of content
        • application/json means body is JSON
        • Tells client how to parse the body

Line 3: Content-Length: 89
        • Content-Length header - Size of body
        • 89 bytes
        • Helps client know when body is complete

Line 4: Date: Mon, 15 Jan 2024 10:00:00 GMT
        • Date header - When response was created
        • GMT timezone
        • Useful for caching and logging

Line 5: Server: nginx/1.18.0
        • Server header - Server software
        • nginx web server, version 1.18.0
        • Optional, but common

Line 6: Cache-Control: no-cache
        • Cache-Control header - Caching instructions
        • no-cache means don't cache this
        • Tells client/browser how to handle caching

Line 7: (blank line)
        • Separates headers from body
        • Required even if no body

Lines 8-12: JSON body
        • The actual data
        • User information in JSON format
        • What the client requested

WHAT ARE HEADERS USED FOR?
--------------------------
Headers provide metadata about the message:

1. REQUEST HEADERS (client to server):
   • Host - Which server/domain
   • User-Agent - Client software
   • Accept - What content types client accepts
   • Authorization - Authentication credentials
   • Content-Type - Type of data in body
   • Content-Length - Size of body
   • Cookie - Stored cookies
   • Referer - Where request came from

2. RESPONSE HEADERS (server to client):
   • Content-Type - Type of content in body
   • Content-Length - Size of body
   • Date - When response was sent
   • Server - Server software
   • Set-Cookie - Cookies to set
   • Cache-Control - Caching instructions
   • Location - Redirect location
   • ETag - Resource version identifier

WHY IS THE BLANK LINE IMPORTANT?
---------------------------------
The blank line is crucial because:
• It separates headers from body
• It tells parsers where headers end
• Without it, the body might be interpreted as headers
• It's required even if there's no body

EXAMPLE WITHOUT BODY:
----------------------
GET /index.html HTTP/1.1
Host: www.example.com

(blank line here - even though no body)

EXAMPLE WITH BODY:
------------------
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "name": "John"
}

(blank line separates headers from JSON body)

HOW ARE HTTP MESSAGES TRANSMITTED?
----------------------------------
HTTP messages are transmitted over TCP/IP:

1. HTTP message is created (text format)
2. Message is sent over TCP connection
3. TCP breaks it into packets
4. Packets travel over IP network
5. Packets reassembled at destination
6. HTTP message reconstructed
7. Server/client processes the message

This is abstracted away - you just create HTTP messages, and the network 
handles the transmission.

PART 2: UNIFORM RESOURCE LOCATORS (URLs)
================================================================================

SECTION 2.1: WHAT IS A URL?
-----------------------------

DEFINITION
----------
A Uniform Resource Locator (URL) is the most popular way to find resources on 
the web. A URL is a string of characters that identifies a specific resource 
on the internet and provides the means to retrieve it.

DETAILED EXPLANATION
--------------------
A URL is essentially an address for a resource on the internet. Just like a 
street address tells you where to find a building, a URL tells your browser 
where to find a web page, image, API endpoint, or any other resource on the 
internet.

WHAT DOES "UNIFORM RESOURCE LOCATOR" MEAN?
-------------------------------------------
Let's break down the term:

1. UNIFORM
   • Means "standardized" or "consistent"
   • All URLs follow the same format
   • Same rules apply everywhere
   • Predictable structure

2. RESOURCE
   • Anything that can be accessed on the web
   • Web pages, images, files, API endpoints
   • Data, services, anything digital

3. LOCATOR
   • Something that tells you where to find something
   • Provides the location/address
   • Points to a specific place

So a URL is: "A standardized way to locate and access resources on the internet."

REAL-WORLD ANALOGY
------------------
Think of a URL like a complete address:

• URL = Complete mailing address
• Scheme = Method of delivery (mail, courier, etc.)
• Domain = City and state
• Path = Street address and apartment number

Example address:
"Send via USPS to: 123 Main Street, Apartment 4B, New York, NY 10001"

Example URL:
"Access via HTTP: www.example.com/images/photo.jpg"

Both tell you exactly where to find something and how to get there.

WHY ARE URLs IMPORTANT?
-----------------------
URLs are crucial because:

1. UNIQUE IDENTIFICATION
   • Each resource has a unique URL
   • No two resources share the same URL
   • Like a fingerprint for web resources

2. NAVIGATION
   • URLs let you navigate the web
   • You can bookmark URLs
   • You can share URLs with others

3. PROGRAMMATIC ACCESS
   • Programs can use URLs to access resources
   • APIs use URLs to identify endpoints
   • Automation tools use URLs

4. SEARCH ENGINE OPTIMIZATION
   • URLs help search engines understand content
   • Well-structured URLs are better for SEO
   • URLs can contain keywords

5. USER-FRIENDLY
   • URLs can be readable
   • Users can understand what a URL points to
   • Easier to remember and share

EXAMPLES OF URLs
----------------
Here are various examples of URLs:

1. WEB PAGE URL:
   https://www.example.com/index.html
   • Points to a web page

2. IMAGE URL:
   https://www.example.com/images/logo.png
   • Points to an image file

3. API ENDPOINT URL:
   https://api.example.com/users/123
   • Points to an API resource

4. FILE URL:
   https://www.example.com/documents/report.pdf
   • Points to a PDF file

5. DIRECTORY URL:
   https://www.example.com/products/
   • Points to a directory (usually shows index.html)

6. URL WITH QUERY PARAMETERS:
   https://www.example.com/search?q=python&page=1
   • Contains search parameters

7. URL WITH FRAGMENT:
   https://www.example.com/page#section1
   • Points to a specific section of a page

SECTION 2.2: BREAKING DOWN URL COMPONENTS
------------------------------------------

DEFINITION
----------
We can break the URL into three parts: First we have the scheme (this is the 
protocol, and for this lab it will always be http://). The internet address or 
Base URL (this will be used to find the location; some examples include 
www.ibm.com and www.gitlab.com). And finally, the route (this is the location 
on the web server, for example: /images/IDSNlogo.png).

DETAILED EXPLANATION
--------------------
Every URL consists of specific components that work together to identify and 
locate a resource. Understanding these components is essential for working 
with web technologies and APIs.

THE THREE MAIN COMPONENTS OF A URL
-----------------------------------
A URL can be broken down into three main parts:

1. SCHEME (Protocol)
   • The method/protocol used to access the resource
   • Always comes first
   • Followed by ://
   • Examples: http://, https://, ftp://

2. INTERNET ADDRESS (Base URL / Domain)
   • The location of the server
   • Identifies which server to connect to
   • Examples: www.ibm.com, www.gitlab.com, api.example.com

3. ROUTE (Path)
   • The specific location on the server
   • The path to the resource
   • Examples: /images/IDSNlogo.png, /index.html, /api/users

VISUAL BREAKDOWN
----------------
Here's how a URL is structured:

┌──────────┐  ┌──────────────────┐  ┌────────────────────┐
│ SCHEME   │  │ INTERNET ADDRESS │  │ ROUTE               │
│ http://  │  │ www.example.com  │  │ /images/logo.png    │
└──────────┘  └──────────────────┘  └────────────────────┘
     │                │                        │
     │                │                        │
  Protocol      Server Location        Resource Path

COMPLETE URL EXAMPLE
---------------------
Let's break down a complete URL:

URL: http://www.ibm.com/images/IDSNlogo.png

BREAKDOWN:
----------
• Scheme: http://
  - Protocol is HTTP
  - Tells browser to use HTTP protocol

• Internet Address: www.ibm.com
  - Server is at IBM's website
  - Domain name identifies the server

• Route: /images/IDSNlogo.png
  - Path on the server
  - Points to IDSNlogo.png in images folder

ANOTHER EXAMPLE
---------------
URL: https://www.gitlab.com/api/v4/projects

BREAKDOWN:
----------
• Scheme: https://
  - Protocol is HTTPS (secure HTTP)
  - Encrypted connection

• Internet Address: www.gitlab.com
  - GitLab's server
  - Domain name

• Route: /api/v4/projects
  - API endpoint path
  - Version 4 API, projects resource

WHAT ARE ADDITIONAL URL COMPONENTS?
------------------------------------
While the three main components are essential, URLs can also include:

1. PORT NUMBER
   • Specifies which port to connect to
   • Default: 80 for HTTP, 443 for HTTPS
   • Example: http://www.example.com:8080

2. QUERY PARAMETERS
   • Additional data after ?
   • Key-value pairs
   • Example: ?name=John&age=30

3. FRAGMENT
   • Points to specific section
   • After #
   • Example: #section1

4. USERNAME/PASSWORD
   • Authentication in URL (rare)
   • Example: http://user:pass@example.com

COMPLETE URL WITH ALL COMPONENTS
---------------------------------
Example: https://user:password@www.example.com:8080/path/to/resource?param1=value1&param2=value2#section

COMPONENTS:
-----------
• Scheme: https://
• Username: user
• Password: password
• Internet Address: www.example.com
• Port: 8080
• Route: /path/to/resource
• Query: ?param1=value1&param2=value2
• Fragment: #section

SECTION 2.3: THE SCHEME (PROTOCOL)
-----------------------------------

DEFINITION
----------
First we have the scheme: this is the protocol, and for this lab it will 
always be http://

DETAILED EXPLANATION
--------------------
The scheme (also called the protocol) is the first part of a URL. It tells 
your browser or application what protocol to use when accessing the resource. 
The scheme determines how the communication will happen.

WHAT IS A PROTOCOL?
-------------------
A protocol is a set of rules that define how communication happens. Think of 
it like:
• Protocol = The language/rules for communication
• Different protocols = Different ways to communicate
• HTTP = One specific protocol for web communication

COMMON SCHEMES (PROTOCOLS)
---------------------------
Here are common URL schemes:

1. HTTP://
   • HyperText Transfer Protocol
   • Standard web protocol
   • Not encrypted
   • Port 80 (default)

2. HTTPS://
   • HTTP Secure
   • Encrypted HTTP
   • Secure communication
   • Port 443 (default)

3. FTP://
   • File Transfer Protocol
   • For file transfers
   • Port 21 (default)

4. FILE://
   • Local file access
   • Access files on your computer
   • Not over network

5. MAILTO:
   • Email links
   • Opens email client
   • Example: mailto:user@example.com

WHY "HTTP://" FOR THIS LAB?
---------------------------
The lab specifies http:// because:
• HTTP is the standard protocol for REST APIs
• Simpler than HTTPS (no encryption complexity)
• Most learning examples use HTTP
• Easier to understand and debug
• Works the same way as HTTPS (just without encryption)

WHAT DOES THE SCHEME DO?
------------------------
The scheme tells the application:

1. WHICH PROTOCOL TO USE
   • http:// = Use HTTP protocol
   • https:// = Use HTTPS protocol
   • ftp:// = Use FTP protocol

2. HOW TO COMMUNICATE
   • Different protocols have different rules
   • HTTP uses request-response model
   • FTP uses file transfer commands

3. WHICH PORT TO USE (default)
   • http:// = Port 80
   • https:// = Port 443
   • Can be overridden with :port

4. WHAT FORMAT TO EXPECT
   • HTTP expects HTTP messages
   • FTP expects FTP commands
   • Each protocol has its own format

EXAMPLES OF SCHEMES IN URLs
----------------------------
1. HTTP URL:
   http://www.example.com
   • Uses HTTP protocol
   • Standard web access

2. HTTPS URL:
   https://www.example.com
   • Uses HTTPS protocol
   • Secure web access

3. FTP URL:
   ftp://files.example.com
   • Uses FTP protocol
   • File transfer

4. FILE URL:
   file:///C:/Users/Documents/file.txt
   • Local file access
   • Not network-based

WHAT HAPPENS IF YOU OMIT THE SCHEME?
-------------------------------------
If you omit the scheme:
• Browser assumes http:// (usually)
• May not work correctly
• Can cause security warnings
• Best practice: Always include scheme

EXAMPLE:
--------
Without scheme: www.example.com
• Browser might assume http://www.example.com
• But it's ambiguous

With scheme: http://www.example.com
• Clear and explicit
• Browser knows exactly what to do

HTTP VS HTTPS
-------------
HTTP (http://):
• Not encrypted
• Data can be intercepted
• Faster (no encryption overhead)
• Port 80
• Good for development/testing

HTTPS (https://):
• Encrypted with SSL/TLS
• Secure communication
• Slightly slower (encryption overhead)
• Port 443
• Required for production (security)

For REST APIs:
• Development: Often use HTTP
• Production: Should use HTTPS
• APIs handling sensitive data: Must use HTTPS

SECTION 2.4: THE INTERNET ADDRESS (BASE URL)
---------------------------------------------

DEFINITION
----------
The internet address or Base URL: this will be used to find the location; 
some examples include www.ibm.com and www.gitlab.com

DETAILED EXPLANATION
--------------------
The internet address (also called the domain name, hostname, or base URL) is 
the part of the URL that identifies which server on the internet you want to 
connect to. It's like the address of a building - it tells you where to go 
to find what you're looking for.

WHAT IS AN INTERNET ADDRESS?
-----------------------------
An internet address is:
• A human-readable name for a server
• Maps to an IP address (like 192.168.1.1)
• Easier to remember than numbers
• Examples: www.ibm.com, www.gitlab.com, api.example.com

REAL-WORLD ANALOGY
------------------
Think of the internet address like a business address:

• Internet Address = Business name and location
• www.ibm.com = "IBM Headquarters, Armonk, NY"
• www.gitlab.com = "GitLab Office, San Francisco, CA"

Just as you need to know the address to visit a business, you need to know 
the internet address to connect to a server.

HOW DOES IT WORK?
-----------------
1. You type www.example.com in your browser
2. Browser asks DNS (Domain Name System): "What's the IP for www.example.com?"
3. DNS responds: "192.0.2.1"
4. Browser connects to 192.0.2.1
5. Browser sends HTTP request to that IP address

This translation happens automatically - you just use the domain name.

COMPONENTS OF AN INTERNET ADDRESS
----------------------------------
An internet address can have multiple parts:

1. SUBDOMAIN
   • Part before the main domain
   • Examples: www, api, blog, mail
   • www.example.com (www is subdomain)

2. DOMAIN NAME
   • The main identifier
   • Examples: ibm, gitlab, example
   • Registered and unique

3. TOP-LEVEL DOMAIN (TLD)
   • The extension
   • Examples: .com, .org, .net, .edu
   • .com is most common

EXAMPLES BREAKDOWN
------------------
1. www.ibm.com
   • Subdomain: www
   • Domain: ibm
   • TLD: .com
   • Full: www.ibm.com

2. www.gitlab.com
   • Subdomain: www
   • Domain: gitlab
   • TLD: .com
   • Full: www.gitlab.com

3. api.example.com
   • Subdomain: api
   • Domain: example
   • TLD: .com
   • Full: api.example.com

4. blog.example.org
   • Subdomain: blog
   • Domain: example
   • TLD: .org
   • Full: blog.example.org

WHAT IS A BASE URL?
-------------------
The base URL is the combination of scheme + internet address:

• Base URL = Scheme + Internet Address
• Example: http://www.ibm.com
• Example: https://api.gitlab.com

The base URL identifies:
• Which protocol to use (http/https)
• Which server to connect to (domain name)
• Everything else (path, query, etc.) comes after

WHY IS IT CALLED "BASE" URL?
----------------------------
It's called "base" because:
• It's the foundation/base of the full URL
• Other parts (path, query) are added to it
• Multiple resources can share the same base URL
• It identifies the server, not a specific resource

EXAMPLES OF BASE URLs
---------------------
1. http://www.ibm.com
   • Base URL for IBM website
   • All IBM pages share this base

2. https://api.gitlab.com
   • Base URL for GitLab API
   • All API endpoints share this base

3. http://localhost:3000
   • Base URL for local development
   • localhost = your own computer
   • 3000 = port number

4. https://www.example.com
   • Base URL for example website
   • Standard web base URL

HOW SERVERS USE BASE URLs
-------------------------
Servers use base URLs to:
• Identify which website/application to serve
• Handle multiple websites on one server (virtual hosting)
• Route requests to the correct application
• Apply security policies per domain

EXAMPLE: VIRTUAL HOSTING
-------------------------
One server can host multiple websites:

• www.example.com → Serves example website
• www.another.com → Serves another website
• api.example.com → Serves API

All on the same physical server, but different base URLs route to different 
applications.

WHAT ABOUT IP ADDRESSES?
------------------------
You can also use IP addresses directly:

• http://192.0.2.1
• http://[2001:db8::1] (IPv6)

But domain names are preferred because:
• Easier to remember
• Can change IP without changing URLs
• More professional
• Better for branding

SECTION 2.5: THE ROUTE (PATH)
------------------------------

DEFINITION
----------
And finally, the route: this is the location on the web server, for example: 
/images/IDSNlogo.png

DETAILED EXPLANATION
--------------------
The route (also called the path) is the part of the URL that specifies the 
exact location of a resource on the server. It comes after the internet 
address and tells the server which specific file, endpoint, or resource you 
want to access.

WHAT IS A ROUTE?
----------------
A route is:
• The path to a specific resource on the server
• Like a file path on your computer
• Tells the server what you're looking for
• Examples: /images/IDSNlogo.png, /index.html, /api/users

REAL-WORLD ANALOGY
------------------
Think of the route like directions within a building:

• Internet Address = Building address
• Route = Room number or file location

Example:
• Building: www.example.com (the address)
• Route: /images/logo.png (go to images folder, get logo.png file)

Just like you need both the building address AND the room number to find 
someone, you need both the internet address AND the route to find a resource.

HOW ROUTES WORK
---------------
Routes work like file paths:

1. / = Root directory
   • Usually maps to index.html
   • The main/home page

2. /images/ = Images directory
   • Contains image files
   • Example: /images/logo.png

3. /api/ = API directory
   • Contains API endpoints
   • Example: /api/users

4. /documents/ = Documents directory
   • Contains document files
   • Example: /documents/report.pdf

EXAMPLES OF ROUTES
------------------
1. /images/IDSNlogo.png
   • Route to an image file
   • In the images folder
   • File name: IDSNlogo.png

2. /index.html
   • Route to the main page
   • In the root directory
   • Default home page

3. /api/users/123
   • Route to an API endpoint
   • API resource: user with ID 123
   • REST API example

4. /products/laptops/dell-xps
   • Route to a product page
   • Nested path structure
   • Hierarchical organization

5. /blog/2024/01/15/my-post
   • Route to a blog post
   • Organized by date
   • SEO-friendly URL

WHAT DOES THE LEADING SLASH MEAN?
----------------------------------
The leading slash (/) is important:
• / = Absolute path from root
• No / = Relative path from current location

Examples:
• /images/logo.png = From server root
• images/logo.png = From current directory

For URLs, you almost always use absolute paths (starting with /).

HOW SERVERS INTERPRET ROUTES
-----------------------------
Servers interpret routes differently:

1. STATIC FILES
   • Route maps directly to file system
   • /images/logo.png → /var/www/images/logo.png
   • Server serves the file

2. DYNAMIC ROUTES (APIs)
   • Route maps to code/function
   • /api/users → handleUsers() function
   • Code generates response

3. ROUTING RULES
   • Server has routing configuration
   • Defines what each route does
   • Can rewrite or redirect routes

EXAMPLE: STATIC FILE SERVING
-----------------------------
Route: /images/IDSNlogo.png

Server process:
1. Receives request for /images/IDSNlogo.png
2. Looks in /var/www/html/images/ directory
3. Finds IDSNlogo.png file
4. Sends file to client

EXAMPLE: API ROUTING
---------------------
Route: /api/users/123

Server process:
1. Receives request for /api/users/123
2. Matches route pattern /api/users/:id
3. Calls getUserById(123) function
4. Function queries database
5. Returns user data as JSON

WHAT ABOUT DEFAULT ROUTES?
---------------------------
When you visit just the domain (www.example.com), the server looks for a 
default file:

• Route: / (root)
• Server looks for: index.html (by default)
• Serves: /index.html automatically

This is why www.example.com and www.example.com/index.html often show the 
same page.

ROUTE EXAMPLES IN CONTEXT
--------------------------
Complete URLs with routes:

1. http://www.ibm.com/images/IDSNlogo.png
   • Scheme: http://
   • Address: www.ibm.com
   • Route: /images/IDSNlogo.png

2. https://api.gitlab.com/api/v4/projects
   • Scheme: https://
   • Address: api.gitlab.com
   • Route: /api/v4/projects

3. http://localhost:3000/api/users/123
   • Scheme: http://
   • Address: localhost:3000
   • Route: /api/users/123

NESTED ROUTES
-------------
Routes can be nested (have multiple levels):

• /api/v1/users/123/posts/456
  - /api = API base
  - /v1 = API version
  - /users = Users resource
  - /123 = User ID
  - /posts = Posts resource
  - /456 = Post ID

This creates a hierarchical structure that's easy to understand.

QUERY PARAMETERS VS ROUTES
--------------------------
Routes identify resources, query parameters provide additional information:

• Route: /api/users/123
  - Identifies: User with ID 123

• Query: /api/users/123?include=posts&fields=name,email
  - Route: /api/users/123 (the resource)
  - Query: ?include=posts&fields=name,email (options/filters)

Both are part of the URL, but serve different purposes.
